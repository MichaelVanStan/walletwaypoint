Create `src/components/calculator/calculator-shell.tsx` with `"use client"` directive.

**Props:** `{ config: CalculatorConfig, computeResults: (values: Record<string, number | string>) => CalculatorResults, params: Record<string, any> }`

The shell is the main calculator engine component. It does NOT contain calculator-specific math -- that is injected via the `computeResults` prop. The `params` prop provides the nuqs parameter definitions for URL state binding. These props will be provided by a CalculatorPageClient wrapper component (created in Plan 06) that imports from the registry.

IMPORTANT: Do NOT create an inline paramRegistry mapping inside CalculatorShell. The params are received as a prop to avoid duplication with registry.ts (Plan 06). This keeps CalculatorShell as a pure UI orchestrator with no knowledge of which calculators exist.

**URL State Management (per CALC-06, nuqs pattern):**
- Call `useQueryStates(params, { history: 'replace', throttleMs: 200 })` to get `[values, setValues]`
- This gives reactive URL-synced state. When any input changes, URL updates after 200ms throttle (per research Pitfall 3)
- Build a `handleChange` function: `(key: string, value: number | string) => setValues({ [key]: value })`

**Computation:**
- Use `useMemo` to compute results from current values: `const results = useMemo(() => computeResults(values), [values, computeResults])`
- For comparison mode: compute `resultsB` from `b_` prefixed values when `values.compare === true`

- When compare is true, render second CalculatorInputs for Scenario B with `b_` prefixed keys and scenarioLabel="Alternative Scenario"
- On mobile (< md), when compare is true, use Tabs component with two tabs: "Your Scenario" and "Alternative Scenario" (per D-11, UI-SPEC)
