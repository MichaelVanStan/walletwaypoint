**1. Refactor `src/app/sitemap.ts`** to use `generateSitemaps` for split sitemaps (per 06-RESEARCH.md Pattern 5 and PSEO-09).

Use the `generateSitemaps` + `sitemap()` pattern to produce multiple sitemap files:

```typescript
import type { MetadataRoute } from 'next';
import { siteConfig } from '@/lib/site-config';
import { calculators, guides, hubs, products, listicles, states, cities, homebuyerPrograms } from '#site/content';

export async function generateSitemaps() {
  return [
    { id: 'core' },
    { id: 'paycheck' },
    { id: 'state-taxes' },
    { id: 'homebuyer' },
    { id: 'city-rent' },
  ];
}

export default async function sitemap(props: {
  id: Promise<string>;
}): Promise<MetadataRoute.Sitemap> {
  const id = await props.id;
  const baseUrl = siteConfig.url;

  switch (id) {
    case 'core':
      return coreUrls(baseUrl);
    case 'paycheck':
      return paycheckUrls(baseUrl);
    case 'state-taxes':
      return stateTaxUrls(baseUrl);
    case 'homebuyer':
      return homebuyerUrls(baseUrl);
    case 'city-rent':
      return cityRentUrls(baseUrl);
    default:
      return [];
  }
}
```

**`coreUrls()`** includes everything from the current sitemap: static pages, calculators (from Velite), guides, hubs, products, listicles, plus new pages:
- `/calculators/paycheck` (main paycheck page)
- `/compare/best` (listicle index)
- All 4 new hub pages

Keep the same `CONTENT_DATES` approach for static page dates. Use `lastUpdated` from Velite data for guides. Use `new Date('2026-03-30')` for new Phase 6 content.

**`paycheckUrls()`** generates 51 entries: one for each `states` entry at `/calculators/paycheck/${state.slug}`. Set `lastModified` to `new Date('2026-03-30')`, `changeFrequency: 'monthly'`, `priority: 0.7`.

**`stateTaxUrls()`** generates 51 entries at `/guides/state-taxes/${state.slug}`.

**`homebuyerUrls()`** generates 51 entries at `/guides/first-time-buyer-programs/${program.slug}`.

**`cityRentUrls()`** generates 25 entries at `/calculators/rent-affordability/${city.slug}`.

This produces URLs like `/sitemap/core.xml`, `/sitemap/paycheck.xml`, etc.

**2. Fix route conflict in `src/app/calculators/[slug]/page.tsx`:**

The `[slug]` route's `generateStaticParams` returns slugs from the Velite `calculators` collection. With the new dedicated directories for `/calculators/paycheck/` and `/calculators/rent-affordability/`, Next.js static segments take precedence over dynamic ones. However, to be explicit and avoid any edge cases:

- In `generateStaticParams`, filter out slugs that have dedicated route directories:
```typescript
export function generateStaticParams() {
  // Exclude calculators that have dedicated route directories with nested sub-routes
  const dedicatedRoutes = ['paycheck', 'rent-affordability'];
  return calculators
    .filter((calc) => !dedicatedRoutes.includes(calc.slug))
    .map((calc) => ({ slug: calc.slug }));
}
```

This prevents Next.js from trying to pre-render `/calculators/paycheck` through the `[slug]` route (since the dedicated `/calculators/paycheck/page.tsx` handles it).

Note: The `rent-affordability` slug IS in the Velite calculators collection (from `content/calculators/rent-affordability.yaml`). If we created a dedicated `/calculators/rent-affordability/` directory with `[city]/page.tsx`, we also need a `page.tsx` in that directory to handle the base route. If Plan 07 did not create `src/app/calculators/rent-affordability/page.tsx`, check whether the existing `[slug]` route still works for `/calculators/rent-affordability`. If there is a conflict, create a simple redirect or re-export page in the new directory.

Actually, simpler approach for rent-affordability: the `[city]` nested route is under `/calculators/rent-affordability/[city]`. The base `/calculators/rent-affordability` is still handled by `[slug]` since the dedicated `rent-affordability` directory has no `page.tsx` at its root (only `[city]/page.tsx`). Next.js should still route `/calculators/rent-affordability` to `[slug]`. Test this. If it works, no filter needed for rent-affordability.
